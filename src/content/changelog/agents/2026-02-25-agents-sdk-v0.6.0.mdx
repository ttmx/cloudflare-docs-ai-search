The latest release of the [Agents SDK](https://github.com/cloudflare/agents) lets you define an Agent and an McpAgent in the same Worker and connect them over RPC ŌĆö no HTTP, no network overhead. It also makes OAuth opt-in for simple MCP connections, hardens the schema converter for production workloads, and ships a batch of `@cloudflare/ai-chat` reliability fixes.

## RPC transport for MCP

You can now connect an Agent to an McpAgent in the same Worker using a Durable Object binding instead of an HTTP URL. The connection stays entirely within the Cloudflare runtime ŌĆö no network round-trips, no serialization overhead.

Pass the Durable Object namespace directly to `addMcpServer`:

<TypeScriptExample>

```ts
import { Agent } from "agents";

export class MyAgent extends Agent {
	async onStart() {
		// Connect via DO binding ŌĆö no HTTP, no network overhead
		await this.addMcpServer("counter", env.MY_MCP);

		// With props for per-user context
		await this.addMcpServer("counter", env.MY_MCP, {
			props: { userId: "user-123", role: "admin" },
		});
	}
}
```

</TypeScriptExample>

The `addMcpServer` method now accepts `string | DurableObjectNamespace` as the second parameter with full TypeScript overloads, so HTTP and RPC paths are type-safe and cannot be mixed.

Key capabilities:

- **Hibernation support** ŌĆö RPC connections survive Durable Object hibernation automatically. The binding name and props are persisted to storage and restored on wake-up, matching the behavior of HTTP MCP connections.
- **Deduplication** ŌĆö Calling `addMcpServer` with the same server name returns the existing connection instead of creating duplicates. Connection IDs are stable across hibernation restore.
- **Smaller surface area** ŌĆö The RPC transport internals have been rewritten and reduced from 609 lines to 245 lines. `RPCServerTransport` now uses `JSONRPCMessageSchema` from the MCP SDK for validation instead of hand-written checks.

:::note
RPC transport is experimental. The API may change based on feedback. Refer to [the tracking issue](https://github.com/cloudflare/agents/issues/565) for updates.
:::

## Optional OAuth for MCP connections

`addMcpServer()` no longer eagerly creates an OAuth provider for every connection. For servers that do not require authentication, a simple call is all you need:

<TypeScriptExample>

```ts
// No callbackHost, no OAuth config ŌĆö just works
await this.addMcpServer("my-server", "https://mcp.example.com");
```

</TypeScriptExample>

If the server responds with a 401, the SDK throws a clear error: `"This MCP server requires OAuth authentication. Provide callbackHost in addMcpServer options to enable the OAuth flow."` The restore-from-storage flow also handles missing callback URLs gracefully, skipping auth provider creation for non-OAuth servers.

## Hardened JSON Schema to TypeScript converter

The schema converter used by `generateTypes()` and `getAITools()` now handles edge cases that previously caused crashes in production:

- **Depth and circular reference guards** ŌĆö Prevents stack overflows on recursive or deeply nested schemas
- **`$ref` resolution** ŌĆö Supports internal JSON Pointers (`#/definitions/...`, `#/$defs/...`, `#`)
- **Tuple support** ŌĆö `prefixItems` (JSON Schema 2020-12) and array `items` (draft-07)
- **OpenAPI 3.0 `nullable: true`** ŌĆö Supported across all schema branches
- **Per-tool error isolation** ŌĆö One malformed schema cannot crash the full pipeline in `generateTypes()` or `getAITools()`
- **Missing `inputSchema` fallback** ŌĆö `getAITools()` falls back to `{ type: "object" }` instead of throwing

## `@cloudflare/ai-chat` fixes

- **Tool denial flow** ŌĆö Denied tool approvals (`approved: false`) now transition to `output-denied` with a `tool_result`, fixing Anthropic provider compatibility. Custom denial messages are supported via `state: "output-error"` and `errorText`.
- **Abort/cancel support** ŌĆö Streaming responses now properly cancel the reader loop when the abort signal fires and send a done signal to the client.
- **Duplicate message persistence** ŌĆö `persistMessages()` now reconciles assistant messages by content and order, preventing duplicate rows when clients resend full history.
- **`requestId` in `OnChatMessageOptions`** ŌĆö Handlers can now send properly-tagged error responses for pre-stream failures.
- **`redacted_thinking` preservation** ŌĆö The message sanitizer no longer strips Anthropic `redacted_thinking` blocks.
- **`/get-messages` reliability** ŌĆö Endpoint handling moved from a prototype `onRequest()` override to a constructor wrapper, so it works even when users override `onRequest` without calling `super.onRequest()`.
- **Client tool APIs undeprecated** ŌĆö `createToolsFromClientSchemas`, `clientTools`, `AITool`, `extractClientToolSchemas`, and the `tools` option on `useAgentChat` are restored for SDK use cases where tools are defined dynamically at runtime.
- **`jsonSchema` initialization** ŌĆö Fixed `jsonSchema not initialized` error when calling `getAITools()` in `onChatMessage`.
